## 问题

### 问题描述

在实际交易场景中，发现了三个关键问题：

1. **Session 读取失败**：午盘操作无法读取早盘的决策和执行记录
   - 早盘的 Session 没有正确记录操作详情
   - 午盘使用 `query_sessions_by_date` 查询不到早盘数据

2. **闭市总结缺失**：16:00 后的 market-close-summary SKILL 执行不完整
   - 没有生成 `summaries/YYYY-MM-DD.md` 总结文件
   - 没有检查并写入 `errors/` 目录的错误记录
   - 错误检查清单被跳过

3. **买卖策略缺失**：出现首次买入直接 ALL IN 的情况
   - 未遵循分段买入策略
   - 未检查仓位管理规则
   - trading-common SKILL 的约束条件未被执行

### 重现步骤

1. 在早盘时段（9:30-11:30）触发 morning-analysis SKILL
2. 执行若干买卖操作
3. 在午盘时段（13:00-14:30）触发 afternoon-trade SKILL
4. 尝试读取早盘 Session → 查询结果为空
5. 在收盘后（16:00+）触发 market-close-summary SKILL
6. 检查 `.msa/knowledge/summaries/` 和 `.msa/knowledge/errors/` 目录
7. 观察到文件未生成

### 错误消息

```
无明确错误消息，但行为表现为：
- 午盘查询早盘 Session 返回空结果
- summaries/ 目录下无当日总结文件
- errors/ 目录下无错误记录
- 买入操作未遵循分段策略，直接买入全部仓位
```

### 影响评估

- **用户影响**：用户无法看到完整的交易记录和复盘总结
- **系统影响**：
  - Session 上下文丢失，午盘无法参考早盘决策
  - 知识库缺失，无法积累经验和教训
  - 风险控制失效，可能造成重大损失
- **业务影响**：
  - 交易策略执行不一致
  - 错误无法被记录和复用
  - 系统可信度降低

## 根本原因

### 调查过程

1. 检查 SKILL 文档的强制性说明
2. 分析模型执行 SKILL 时的行为
3. 对比设计文档与实际执行

### 根本原因

**问题1：Session 读取失败**
- 早盘 Session 的记录不够详细，关键决策信息未记录
- 可能是模型在执行时跳过了"记录到 Session"的步骤

**问题2：闭市总结缺失**
- SKILL 文档中的"强制"步骤不够强调
- 模型可能认为某些步骤是可选的
- 缺少明确的"必须调用"指令

**问题3：买卖策略缺失**
- trading-common SKILL 作为参考文档，约束力不足
- 早午盘 SKILL 中对仓位管理的引用不够明确
- 缺少强制性的检查步骤

## 解决方案

### 解决方案方法

通过强化 SKILL 文档的措辞和格式，提高模型对强制步骤的执行率：

1. 使用 `⚠️【强制前置】` 标记必须首先执行的步骤
2. 使用 `**必须**` 和 `MUST` 强调强制性
3. 添加执行检查表，明确列出必须完成的步骤
4. 在约束条件中使用更强的警告语言

## 经验教训

### 出了什么问题

- SKILL 文档的强制性表达不够明确
- 模型可能将"必须"理解为建议
- 缺少明确的执行验证机制

### 学到了什么

- 使用视觉标记（⚠️）能显著提高注意力
- 多重强调（中文+英文）比单一语言更有效
- 执行检查表能帮助模型自我验证

### 预防措施

1. 为所有关键 SKILL 添加强制前置标记
2. 使用统一格式：`⚠️【强制前置】` + `**必须**` + `MUST`
3. 添加执行检查表作为自我验证工具
4. 定期审查 SKILL 执行日志，检查步骤遗漏

## 可复用模式

**模式名称**：强制步骤标记模式
**模式ID**：p006

**描述**：
在 AI Agent 的 Skill/提示词文档中，使用多层次的强调机制确保关键步骤被执行。

**适用场景**：
- 安全关键的操作（如交易执行）
- 合规要求的检查
- 不可逆操作前的验证
- 知识积累的关键步骤

**实现**：
```markdown
## ⚠️【强制前置】- 不执行将导致流程失败

**第一步 MUST 调用 `xxx` 工具！**

⚠️ 警告说明...

## 执行检查表

□ 已执行步骤1
□ 已执行步骤2
□ 已执行步骤3
```

**收益**：
- 提高关键步骤的执行率
- 减少人为错误
- 便于事后审计
- 建立执行标准

## 反模式

**反模式名称**：弱语言表达强制要求
**反模式ID**：ap003

**问题所在**：
使用弱语言（如"应该"、"建议"、"可以考虑"）表达必须执行的操作，导致模型将其视为可选。

**避免**：
```markdown
// 不要这样做
- 建议先读取错误记录
- 可以考虑分段买入
- 应该生成总结文件

// 应该这样做
- ⚠️ **必须** 先读取错误记录
- ⚠️ **禁止** 首次买入超过计划仓位的 50%
- ⚠️ **必须** 生成总结文件
```
