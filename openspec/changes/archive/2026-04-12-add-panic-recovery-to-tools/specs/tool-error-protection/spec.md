# 规格：tool-error-protection

> 📚 **知识上下文**：基于错误处理策略模式 #003，避免无 panic recovery 的反模式

统一的工具错误保护机制，防止 panic 中断会话。

## 增加需求

### 需求：R-1 Panic Recovery 保护

所有工具入口函数必须包含 panic recovery 机制，确保内部异常不会导致会话中断。

#### 场景：工具内部发生 panic

- **当** 工具执行过程中发生 panic
- **那么** panic 被捕获并转换为错误 JSON 响应
- **那么** 会话继续运行，不会崩溃
- **那么** 完整的 panic 信息被记录到日志

> 📚 **历史问题参考**：知识库反模式 - 无 panic recovery 机制

### 需求：R-2 统一辅助函数

提供统一的 `SafeExecute` 辅助函数，减少重复代码。

#### 场景：使用 SafeExecute 包装工具函数

- **当** 工具入口函数使用 SafeExecute 包装
- **那么** 自动获得 panic recovery 保护
- **那么** 错误被统一格式化为 JSON
- **那么** 日志记录包含工具名称和参数

### 需求：R-3 错误 JSON 格式

Panic 恢复后返回的错误 JSON 必须符合现有 `ToolResult` 格式。

#### 场景：返回 panic 错误响应

- **当** panic 被捕获
- **那么** 返回 `{"success": false, "error_msg": "工具内部错误: <panic message>"}`
- **那么** LLM 能够理解错误并继续对话

### 需求：R-4 日志记录

所有 panic 必须记录完整的日志信息，便于调试。

#### 场景：记录 panic 日志

- **当** panic 被捕获
- **那么** 记录 ERROR 级别日志
- **那么** 日志包含工具名称、参数、panic 值、堆栈信息
- **那么** 使用 `logrus` 结构化日志

> 📚 **历史问题参考**：基于错误处理策略模式 #003

### 需求：R-5 向后兼容

变更必须保持向后兼容，不影响现有工具的行为。

#### 场景：工具正常执行

- **当** 工具执行成功（无 panic）
- **那么** 返回结果与变更前完全一致
- **那么** 函数签名不变

#### 场景：工具返回错误（非 panic）

- **当** 工具返回 error（正常错误处理）
- **那么** 错误处理逻辑与变更前一致
- **那么** 返回的 JSON 格式与变更前一致

### 需求：R-6 覆盖范围

所有工具入口函数都必须添加保护。

#### 场景：高风险工具优先

- **当** 涉及 HTML 解析、网络请求的工具（search, fetcher）
- **那么** 优先添加保护（已知问题风险高）

#### 场景：低风险工具覆盖

- **当** 其他工具（stock, finance, skill, todo, knowledge）
- **那么** 同样添加保护（防御性编程）

> 📚 **历史问题参考**：问题 #005 Google CAPTCHA 检测、问题 #004 浏览器超时

### 需求：R-7 广播消息

Panic 发生时必须广播工具结束消息，保持 UI 一致性。

#### 场景：广播 panic 错误

- **当** panic 被捕获
- **那么** 调用 `message.BroadcastToolEnd` 通知 UI
- **那么** 用户能看到工具失败的提示
