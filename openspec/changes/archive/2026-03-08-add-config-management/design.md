# 设计：add-config-management

## 上下文

### 当前状态

MSA 项目目前拥有完整的配置系统后端（`pkg/config/local_config.go`），支持：
- 配置文件的读写（`~/.msa/msa_config.json`）
- 日志系统的初始化
- 配置缓存机制

但缺少：
- 用户友好的配置接口
- 环境变量支持
- 命令行参数配置
- 配置验证机制
- Provider 注册表

### 约束条件

1. **框架约束**：TUI 必须使用项目已有的 `bubbletea` 框架
2. **兼容性约束**：必须保持向后兼容，现有配置文件无需修改
3. **依赖约束**：不引入新的外部依赖
4. **扩展性约束**：Provider 系统需要支持未来添加新的提供商

### 利益相关者

- **终端用户**：需要简单易用的配置方式
- **开发者**：需要易于扩展的 Provider 系统
- **运维**：需要支持环境变量配置以适应不同部署环境

---

> 📚 **相关历史问题**：知识库中无直接相关配置问题，但模式 #003（错误处理和重试策略）适用于本设计

## 目标 / 非目标

**目标：**

1. 提供交互式 TUI 配置界面（`msa config`）
2. 支持命令行参数配置（`--config`）
3. 支持环境变量配置
4. 实现 Provider 注册表和验证系统
5. 实现配置验证机制
6. 建立配置优先级系统

**非目标：**

1. 不实现配置文件的动态热重载（需要重启程序）
2. 不实现配置文件加密（API Key 明文存储在用户目录）
3. 不实现配置版本迁移（首次实现，无旧版本）
4. 不实现远程配置中心

## 推荐模式

### 错误处理和重试策略模式

**模式ID**：p003
**来源问题**：知识库 #003

**描述**：
配置操作必须具有清晰的错误处理，用户输入验证应该提供友好的错误提示，配置保存失败时应该有明确的反馈。

**适用场景**：
- 所有配置 I/O 操作（读、写）
- 用户输入验证
- 配置合并和优先级处理

**实现要点**：
1. 配置加载失败不应阻止程序启动
2. 验证错误应分组显示，按严重程度排序
3. 保存失败时应保留用户输入，允许重试
4. 错误消息应包含上下文和修复建议

### 构建器模式（配置合并）

**描述**：
使用分层构建器模式实现配置优先级合并，从低到高逐层应用配置。

**适用场景**：
- 多源配置合并
- 配置优先级处理
- 部分配置覆盖

**实现**：
```go
// 配置构建器
type ConfigBuilder struct {
    defaultCfg  *LocalStoreConfig
    fileCfg     *LocalStoreConfig
    envCfg      *LocalStoreConfig
    cliCfg      *LocalStoreConfig
}

func (b *ConfigBuilder) Build() *LocalStoreConfig {
    result := b.defaultCfg

    // 按优先级从低到高合并
    result = b.merge(result, b.fileCfg)
    result = b.merge(result, b.envCfg)
    result = b.merge(result, b.cliCfg)

    return result
}

func (b *ConfigBuilder) merge(base, override *LocalStoreConfig) *LocalStoreConfig {
    // 只覆盖非零值
    if override.APIKey != "" {
        base.APIKey = override.APIKey
    }
    // ... 其他字段
    return base
}
```

## 避免的反模式

### 静默失败反模式

**反模式ID**：a001
**来源模式**：p003（错误处理和重试策略）

**问题所在**：
配置操作失败时静默忽略，使用默认值而不通知用户，导致用户不知道配置未生效。

**影响**：
- 用户以为是配置问题，实际是加载失败
- 难以排查配置相关故障
- 用户体验差

**不要这样做**：
```go
// 不要这样做 - 静默失败
cfg, _ := loadConfig() // 忽略错误
useConfig(cfg)
```

**应该这样做**：
```go
// 应该这样做 - 明确的错误处理
cfg, err := loadConfig()
if err != nil {
    log.Warnf("配置加载失败，使用默认配置: %v", err)
    if.isUserInteractive() {
        notifyUser("配置文件损坏，已使用默认配置。运行 'msa config' 修复。")
    }
    cfg = getDefaultConfig()
}
useConfig(cfg)
```

### 硬编码验证反模式

**问题所在**：
将验证规则硬编码在多个地方，导致维护困难和规则不一致。

**影响**：
- 添加新 Provider 需要修改多处代码
- 验证规则不一致
- 难以扩展

**应该这样做**：
将验证规则集中在 `ProviderRegistry` 中，TUI 和配置系统都从这里获取规则。

## 设计决策

### 决策 1：使用 Bubbletea 实现 TUI

**选择**：使用项目已有的 `bubbletea` 框架实现 TUI

**理由**：
- 项目已依赖 `bubbletea`（用于 chat TUI）
- 可以复用现有的样式和组件
- 团队熟悉该框架

**替代方案及拒绝原因**：
- `survey`：更简单但功能有限，难以实现复杂的验证反馈
- `termui`：项目未依赖，引入新依赖

### 决策 2：Provider 注册表使用全局 map

**选择**：使用全局 `ProviderRegistry` map 存储 Provider 元信息

**理由**：
- 简单直接，易于扩展
- 支持运行时添加新 Provider
- 与现有的 `providerMap` 模式一致

**数据结构**：
```go
type ProviderInfo struct {
    ID            LlmProvider
    DisplayName   string
    Description   string
    DefaultBaseURL string
    KeyPrefix     string
}

var ProviderRegistry = map[LlmProvider]ProviderInfo{
    Siliconflow: { ... },
}
```

### 决策 3：配置优先级使用后合并策略

**选择**：在程序启动时统一合并所有配置源

**理由**：
- 简单可靠，一次性处理
- 配置缓存生效后性能最佳
- 易于理解和调试

**替代方案及拒绝原因**：
- 惰性合并（每次读取时）：增加复杂度，性能稍差

### 决策 4：验证规则使用 ProviderKeyPrefix

**选择**：API Key 验证使用 Provider 定义的 `KeyPrefix` 进行前缀检查

**理由**：
- 灵活支持不同 Provider 的 Key 格式
- 验证规则与 Provider 元信息集中管理
- 允许为空（跳过验证）

**风险**：某些 Provider 可能没有统一前缀
**缓解**：允许 `KeyPrefix` 为空字符串，跳过前缀验证

### 决策 5：切换 Provider 时清空敏感配置

**选择**：切换 Provider 时自动清空 API Key 和 Base URL

**理由**：
- 避免 API Key 被错误地发送给错误的 Provider
- 提示用户重新配置
- 提高安全性

**用户体验考虑**：
- 自动填充新 Provider 的默认 Base URL
- 在 TUI 中显示提示消息

## 风险 / 权衡

### 风险 1：API Key 明文存储

**风险**：API Key 以明文形式存储在用户目录的 JSON 文件中

**缓解措施**：
- 文件权限设置为 `0644`（仅用户可读写）
- 文档中说明安全注意事项
- 未来可考虑添加密钥加密选项

### 风险 2：配置验证过于严格

**风险**：严格的验证可能阻止某些特殊但有效的配置

**缓解措施**：
- 提供"强制保存"选项允许用户跳过验证
- 使用警告而非错误来提示可能的问题
- 记录详细的验证失败原因

### 风险 3：TUI 复杂度增加

**风险**：TUI 功能增多可能导致代码复杂、难以维护

**缓解措施**：
- 将 TUI 拆分为多个独立组件
- 使用清晰的状态管理
- 编写充分的测试

### 风险 4：Provider 扩展性

**风险**：如果 Provider 注册表设计不当，未来添加新 Provider 可能困难

**缓解措施**：
- 使用注册表模式，添加新 Provider 只需注册
- 不在 TUI 中硬编码 Provider 列表
- 从 `providerMap` 动态获取可用 Provider

### 权衡 1：实时验证 vs 延迟验证

**选择**：使用延迟验证（用户停止输入 500ms 后）

**权衡**：
- 实时验证：反馈即时，但可能在输入过程中显示错误
- 延迟验证：不打断输入，但反馈稍有延迟

### 权衡 2：配置热重载 vs 重启生效

**选择**：配置修改需要重启程序生效

**权衡**：
- 热重载：用户体验更好，但实现复杂，可能有并发问题
- 重启生效：实现简单可靠，但用户体验稍差
