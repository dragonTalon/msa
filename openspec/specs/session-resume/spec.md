# 规格：session-resume

会话恢复能力，通过 `--resume` 参数加载历史会话继续对话。

> 📚 **知识上下文**：参考 `error-handling-strategy` 处理文件解析错误，提供清晰的错误信息。

## 增加需求

### 需求：命令行参数解析

系统支持 `--resume` 参数指定要恢复的会话。

#### 场景：参数格式
- **当** 用户提供 `--resume` 参数时
- **那么** 参数格式为 `YYYY-MM-DD_{uuid前8位}` 或完整 UUID
- **那么** 解析出日期和 UUID 用于定位文件

#### 场景：参数缺失
- **当** 用户提供 `--resume` 但未指定会话 ID 时
- **那么** 系统输出错误提示："请指定要恢复的会话 ID，格式：YYYY-MM-DD_uuid"

### 需求：会话文件定位

系统根据参数定位并加载会话文件。

#### 场景：文件存在
- **当** 指定的会话文件存在时
- **那么** 读取文件内容
- **那么** 解析 frontmatter 获取会话元数据

#### 场景：文件不存在
- **当** 指定的会话文件不存在时
- **那么** 输出错误："会话不存在：{会话 ID}"
- **那么** 程序退出码为 1

### 需求：历史消息加载

系统从会话文件解析历史消息并注入到对话上下文。

#### 场景：解析用户消息
- **当** 解析到 `## 👤 用户` 标题时
- **那么** 将后续内容解析为用户消息
- **那么** 消息角色设为 `RoleUser`

#### 场景：解析助手消息
- **当** 解析到 `## 🤖 MSA` 标题时
- **那么** 将后续内容解析为助手消息
- **那么** 消息角色设为 `RoleAssistant`

#### 场景：忽略其他内容
- **当** 解析到非用户/助手标题时
- **那么** 跳过该内容

#### 场景：历史消息注入
- **当** 解析完成后
- **那么** 历史消息注入到 `history` 数组
- **那么** 保持原始消息顺序

### 需求：恢复模式行为

恢复会话后，系统行为与正常会话一致，但使用原有会话 ID。

#### 场景：TUI 恢复模式
- **当** 使用 `msa --resume {id}` 启动 TUI 时
- **那么** 加载历史消息到对话历史
- **那么** 使用原有 UUID 继续会话
- **那么** 新消息追加到原会话文件

#### 场景：显示恢复信息
- **当** 恢复会话成功时
- **那么** 欢迎信息显示"已恢复会话：{uuid前8位}"
- **那么** 显示历史消息数量

#### 场景：恢复后 clear
- **当** 用户在恢复的会话中执行 `clear` 命令
- **那么** 清空对话历史
- **那么** 生成新的 UUID
- **那么** 开始新会话

### 需求：文件解析错误处理

解析失败时提供清晰的错误信息。

#### 场景：文件格式错误
- **当** 会话文件格式不正确时
- **那么** 输出错误："会话文件格式错误：{原因}"
- **那么** 程序退出码为 1

#### 场景：frontmatter 缺失
- **当** 会话文件缺少 frontmatter 时
- **那么** 输出错误："会话文件缺少元数据"
- **那么** 程序退出码为 1
