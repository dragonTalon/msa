# 规格：transaction-recording

> **能力**: 交易记录
> **目的**: 系统必须提供完整的交易记录能力，包括买卖订单的创建、状态跟踪和查询。

## 需求：创建交易记录

系统必须支持创建新的交易记录。

### 场景：记录买入订单
- **当** 用户提交买入订单
- **那么** 系统创建交易记录
- **并且** 类型为 BUY，状态为 PENDING
- **并且** 包含：股票代码、股票名称、数量、价格、金额、手续费

### 场景：记录卖出订单
- **当** 用户提交卖出订单
- **那么** 系统创建交易记录
- **并且** 类型为 SELL，状态为 PENDING
- **并且** 卖出订单不锁定账户金额

### 场景：关联账户
- **当** 创建交易记录时
- **那么** 必须关联到对应的账户 ID
- **并且** 外键约束确保引用完整性

---

## 需求：交易状态管理

系统必须支持交易状态的完整生命周期管理。

### 场景：状态流转 - 买入订单

买入订单的状态流转：
- PENDING（挂单中）→ FILLED（已成交）/ CANCELLED（已撤销）/ REJECTED（已拒绝）
- 部分成交时：PENDING → OBSOLETE（原记录）+ FILLED（已成交部分）+ PENDING（剩余部分）

### 场景：状态流转 - 卖出订单

卖出订单的状态流转：
- PENDING（挂单中）→ FILLED（已成交）/ CANCELLED（已撤销）

### 场景：订单被拒绝
- **当** 买入订单余额不足
- **那么** 系统创建状态为 REJECTED 的交易记录
- **并且** note 字段包含拒绝原因（如"余额不足"、"持仓不足"）

### 场景：部分成交处理
- **当** 订单部分成交
- **那么** 原记录状态更新为 OBSOLETE
- **并且** 创建新的 FILLED 记录（已成交部分）
- **并且** 创建新的 PENDING 记录（剩余部分）
- **并且** 新记录通过 parent_id 关联原记录

---

## 需求：查询交易记录

系统必须支持多维度查询交易记录。

### 场景：按账户查询
- **当** 查询某账户的所有交易
- **那么** 系统返回该账户的所有交易记录
- **并且** 按创建时间倒序排列

### 场景：按股票查询
- **当** 查询某只股票的所有交易
- **那么** 系统返回该股票的所有交易记录
- **并且** 跨账户聚合

### 场景：按类型查询
- **当** 查询所有买入或卖出交易
- **那么** 系统返回对应类型的交易记录

### 场景：按状态查询
- **当** 查询特定状态的交易（如 PENDING、FILLED）
- **那么** 系统返回该状态的所有交易记录

### 场景：查询子交易
- **当** 查询部分成交的子记录
- **那么** 系统根据 parent_id 返回所有子交易
- **并且** 按创建时间正序排列

---

## 需求：交易金额计算

系统必须正确计算和记录交易相关金额。

### 场景：计算交易金额
- **当** 创建交易记录时
- **那么** 交易金额 = 数量 × 价格
- **并且** 以分为单位存储（整数）

### 场景：获取总支出
- **当** 查询买入交易的总成本
- **那么** 总支出 = 交易金额 + 手续费

### 场景：获取总收入
- **当** 查询卖出交易的总收入
- **那么** 总收入 = 交易金额 - 手续费

---

## 需求：交易记录完整性

系统必须确保交易记录的完整性和一致性。

### 场景：记录创建时间
- **当** 创建交易记录时
- **那么** 系统自动记录创建时间
- **并且** 创建时间不可修改

### 场景：记录备注信息
- **当** 订单被拒绝或特殊处理时
- **那么** note 字段记录原因或备注信息
