# 规格：amount-locking

## Purpose

系统 SHALL 提供金额锁定机制，确保挂单订单的资金安全，支持订单成交和撤销时的资金处理。

## Requirements

### Requirement: 买入订单锁定金额

系统 SHALL 在买入订单提交时锁定相应金额，确保资金不被重复使用。

#### Scenario: 正常买入订单
- **WHEN** 用户提交买入订单且账户余额充足
- **THEN** 系统在事务中执行：
  1. 检查账户可用金额是否充足
  2. 创建 PENDING 状态的交易记录
  3. 从可用金额扣除订单总金额（交易金额 + 手续费）
  4. 将扣除的金额加入锁定金额
- **AND** 所有操作原子性执行
- **AND** 返回交易记录 ID

#### Scenario: 余额不足拒绝
- **WHEN** 买入订单金额超过账户可用金额
- **THEN** 系统创建 REJECTED 状态的交易记录
- **AND** note 字段说明原因（如"余额不足"）
- **AND** 不修改账户金额

#### Scenario: 订单金额计算
- **WHEN** 计算需要锁定的金额时
- **THEN** 锁定金额 = 交易金额 + 手续费
- **AND** 交易金额 = 数量 × 价格

---

### Requirement: 卖出订单不锁定

卖出订单 SHALL NOT 锁定金额，但需要验证持仓充足。

#### Scenario: 正常卖出订单
- **WHEN** 用户提交卖出订单且持仓充足
- **THEN** 系统创建 PENDING 状态的交易记录
- **AND** 不锁定账户金额
- **AND** 验证持仓数量足够

#### Scenario: 持仓不足拒绝
- **WHEN** 卖出订单数量超过持仓数量
- **THEN** 系统创建 REJECTED 状态的交易记录
- **AND** note 字段说明原因（如"持仓不足"）

---

### Requirement: 订单成交处理

系统 SHALL 根据订单类型正确处理成交时的金额变化。

#### Scenario: 买入订单成交
- **WHEN** 买入订单成交
- **THEN** 系统在事务中执行：
  1. 更新交易状态为 FILLED
  2. 减少锁定金额（已成交部分）
- **AND** 可用金额保持不变（已在提交时扣除）

#### Scenario: 卖出订单成交
- **WHEN** 卖出订单成交
- **THEN** 系统在事务中执行：
  1. 更新交易状态为 FILLED
  2. 增加可用金额（成交金额 - 手续费）
- **AND** 不涉及锁定金额

---

### Requirement: 订单撤销处理

系统 SHALL 正确处理订单撤销时的金额返还。

#### Scenario: 撤销买入订单
- **WHEN** 买入订单被撤销
- **THEN** 系统在事务中执行：
  1. 更新交易状态为 CANCELLED
  2. 锁定金额返还到可用金额
  3. 确保返还金额等于原锁定金额

#### Scenario: 撤销卖出订单
- **WHEN** 卖出订单被撤销
- **THEN** 系统更新交易状态为 CANCELLED
- **AND** 不调整账户金额（卖出订单未锁定金额）

---

### Requirement: 部分成交处理

系统 SHALL 支持订单部分成交时的锁定金额调整。

#### Scenario: 买入订单部分成交
- **WHEN** 买入订单部分成交
- **THEN** 系统在事务中执行：
  1. 将原交易记录状态更新为 OBSOLETE
  2. 创建已成交部分的 FILLED 记录
  3. 创建剩余部分的 PENDING 记录
  4. 调整锁定金额为剩余部分的金额
  5. 新记录通过 parent_id 关联原记录

#### Scenario: 计算剩余锁定金额
- **WHEN** 调整锁定金额时
- **THEN** 剩余锁定金额 = 原锁定金额 - 已成交部分的锁定金额
- **AND** 已成交部分的锁定金额 = 已成交数量 × 价格 + 对应手续费

---

### Requirement: 事务一致性

系统 MUST 确保所有金额更新操作的事务一致性。

#### Scenario: 原子性保证
- **WHEN** 执行金额锁定、解锁、调整操作时
- **THEN** 操作必须在事务中执行
- **AND** 使用 defer 确保异常时自动回滚

#### Scenario: 并发安全
- **WHEN** 多个订单同时处理同一账户
- **THEN** 数据库必须提供并发控制
- **AND** 防止金额不一致
- **AND** 按顺序串行处理金额更新
