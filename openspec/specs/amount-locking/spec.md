# 规格：amount-locking

> **能力**: 金额锁定
> **目的**: 系统必须提供金额锁定机制，确保挂单订单的资金安全，支持订单成交和撤销时的资金处理。

## 需求：买入订单锁定金额

系统必须在买入订单提交时锁定相应金额，确保资金不被重复使用。

### 场景：正常买入订单
- **当** 用户提交买入订单且账户余额充足
- **那么** 系统在事务中执行：
  1. 检查账户可用金额是否充足
  2. 创建 PENDING 状态的交易记录
  3. 从可用金额扣除订单总金额（交易金额 + 手续费）
  4. 将扣除的金额加入锁定金额
- **并且** 所有操作原子性执行
- **并且** 返回交易记录 ID

### 场景：余额不足拒绝
- **当** 买入订单金额超过账户可用金额
- **那么** 系统创建 REJECTED 状态的交易记录
- **并且** note 字段说明原因（如"余额不足"）
- **并且** 不修改账户金额

### 场景：订单金额计算
- **当** 计算需要锁定的金额时
- **那么** 锁定金额 = 交易金额 + 手续费
- **并且** 交易金额 = 数量 × 价格

---

## 需求：卖出订单不锁定

卖出订单不需要锁定金额，但需要验证持仓充足。

### 场景：正常卖出订单
- **当** 用户提交卖出订单且持仓充足
- **那么** 系统创建 PENDING 状态的交易记录
- **并且** 不锁定账户金额
- **并且** 验证持仓数量足够

### 场景：持仓不足拒绝
- **当** 卖出订单数量超过持仓数量
- **那么** 系统创建 REJECTED 状态的交易记录
- **并且** note 字段说明原因（如"持仓不足"）

---

## 需求：订单成交处理

系统必须根据订单类型正确处理成交时的金额变化。

### 场景：买入订单成交
- **当** 买入订单成交
- **那么** 系统在事务中执行：
  1. 更新交易状态为 FILLED
  2. 减少锁定金额（已成交部分）
- **并且** 可用金额保持不变（已在提交时扣除）

### 场景：卖出订单成交
- **当** 卖出订单成交
- **那么** 系统在事务中执行：
  1. 更新交易状态为 FILLED
  2. 增加可用金额（成交金额 - 手续费）
- **并且** 不涉及锁定金额

---

## 需求：订单撤销处理

系统必须正确处理订单撤销时的金额返还。

### 场景：撤销买入订单
- **当** 买入订单被撤销
- **那么** 系统在事务中执行：
  1. 更新交易状态为 CANCELLED
  2. 锁定金额返还到可用金额
  3. 确保返还金额等于原锁定金额

### 场景：撤销卖出订单
- **当** 卖出订单被撤销
- **那么** 系统更新交易状态为 CANCELLED
- **并且** 不调整账户金额（卖出订单未锁定金额）

---

## 需求：部分成交处理

系统必须支持订单部分成交时的锁定金额调整。

### 场景：买入订单部分成交
- **当** 买入订单部分成交
- **那么** 系统在事务中执行：
  1. 将原交易记录状态更新为 OBSOLETE
  2. 创建已成交部分的 FILLED 记录
  3. 创建剩余部分的 PENDING 记录
  4. 调整锁定金额为剩余部分的金额
  5. 新记录通过 parent_id 关联原记录

### 场景：计算剩余锁定金额
- **当** 调整锁定金额时
- **那么** 剩余锁定金额 = 原锁定金额 - 已成交部分的锁定金额
- **并且** 已成交部分的锁定金额 = 已成交数量 × 价格 + 对应手续费

---

## 需求：事务一致性

系统必须确保所有金额更新操作的事务一致性。

### 场景：原子性保证
- **当** 执行金额锁定、解锁、调整操作时
- **那么** 操作必须在事务中执行
- **并且** 使用 defer 确保异常时自动回滚

### 场景：并发安全
- **当** 多个订单同时处理同一账户
- **那么** 数据库必须提供并发控制
- **并且** 防止金额不一致
- **或者** 按顺序串行处理金额更新
