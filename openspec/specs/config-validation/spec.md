# 规格：config-validation

配置验证能力，确保所有配置项的有效性。

## Purpose

提供全面的配置验证机制，确保用户输入的配置值符合格式要求，并在配置错误时提供友好的错误提示和修复建议。

## Requirements

### 需求：API Key 验证

系统 SHALL 验证用户输入的 API Key 格式。

#### 场景：非空验证
- **当** 用户提交 API Key 配置
- **并且** API Key 为空字符串
- **那么** 系统应显示错误："API Key 不能为空"

#### 场景：前缀验证
- **当** Provider 定义了 API Key 前缀
- **并且** 用户输入的 API Key 不以该前缀开头
- **那么** 系统应显示错误："API Key 应该以 {prefix} 开头"

#### 场景：长度验证
- **当** 用户输入的 API Key 少于 10 个字符
- **那么** 系统应显示警告："API Key 长度可能不正确"

#### 场景：跳过验证警告
- **当** API Key 验证失败
- **那么** 系统应提供"强制保存"选项
- **并且** 显示警告："跳过验证可能导致 API 调用失败"

### 需求：URL 验证

系统 SHALL 验证 Base URL 的格式和可访问性。

#### 场景：格式验证
- **当** 用户输入 Base URL
- **并且** 不是有效的 URL 格式
- **那么** 系统应显示错误："请输入有效的 URL（如 https://api.example.com）"

#### 场景：协议要求
- **当** 用户输入的 URL 不以 http:// 或 https:// 开头
- **那么** 系统应显示错误："URL 必须以 http:// 或 https:// 开头"

#### 场景：默认 URL 填充
- **当** 用户选择 Provider
- **那么** 系统应自动填充 Provider 的默认 Base URL
- **并且** 用户可以修改

#### 场景：空 URL 处理
- **当** Base URL 为空
- **并且** Provider 有默认 URL
- **那么** 系统应使用默认 URL
- **并且** 不显示错误

### 需求：日志级别验证

系统 SHALL 验证日志级别配置的有效性。

#### 场景：有效级别枚举
- **当** 用户设置日志级别
- **那么** 系统应接受以下值（不区分大小写）：
  - `debug`
  - `info`
  - `warn` 或 `warning`
  - `error`
  - `fatal`
  - `panic`

#### 场景：无效级别
- **当** 用户输入无效的日志级别
- **那么** 系统应显示错误："无效的日志级别，支持：debug, info, warn, error, fatal, panic"

#### 场景：大小写转换
- **当** 用户输入如 `DEBUG` 或 `Info` 等混合大小写
- **那么** 系统应转换为小写存储
- **并且** 不显示错误

### 需求：日志路径验证

系统 SHALL 验证日志文件路径的有效性和可写性。

#### 场景：目录存在性
- **当** 用户设置日志文件路径
- **并且** 指定的目录不存在
- **那么** 系统应尝试创建目录
- **并且** 如果创建失败，显示错误

#### 场景：文件可写性
- **当** 用户设置日志文件路径
- **并且** 指定的路径不可写
- **那么** 系统应显示错误："无法写入日志文件，请检查路径权限"

#### 场景：绝对路径
- **当** 用户输入相对路径
- **那么** 系统应转换为相对于用户主目录的路径
- **并且** 在显示时展示完整路径

#### 场景：路径展开
- **当** 路径包含 `~` 或环境变量如 `$HOME`
- **那么** 系统应展开为完整路径
- **并且** 存储展开后的路径

### 需求：Provider 验证

系统 SHALL 验证 Provider 配置的有效性。

#### 场景：Provider 存在性
- **当** 用户设置 Provider
- **并且** Provider 未在 `ProviderRegistry` 中注册
- **那么** 系统应显示错误："未知的 Provider: {name}"

#### 场景：已注册 Provider
- **当** 用户从 TUI 选择器中选择 Provider
- **那么** 系统应保证选择的 Provider 已注册
- **并且** 不需要额外验证

### 需求：验证错误处理

系统 MUST 正确处理多个验证错误并给出友好提示。
- **当** 配置保存时存在多个验证错误
- **那么** 系统应列出所有错误
- **并且** 高亮显示每个错误的配置项
- **并且** 按严重程度排序显示

#### 场景：错误恢复
- **当** 验证失败后
- **那么** 系统应保持用户输入的值
- **并且** 允许用户修改后重试
- **并且** 提供"取消"选项返回主界面

#### 场景：警告级别
- **当** 配置项有问题但不影响使用
- **那么** 系统应显示警告而非错误
- **并且** 允许用户选择是否继续
- **例如**：API Key 长度较短

### 需求：实时验证

系统 SHALL 在用户输入时提供即时反馈。

#### 场景：输入时验证
- **当** 用户在 TUI 中输入配置值
- **那么** 系统应在用户停止输入后验证
- **并且** 在输入框下方显示验证状态

#### 场景：验证状态指示
- **当** 配置值验证通过
- **那么** 显示绿色对勾或"✓ 有效"
- **当** 配置值验证失败
- **那么** 显示红色叉号或"✗ 无效"及错误原因

#### 场景：延迟验证
- **当** 用户正在输入
- **那么** 系统应等待用户停止输入（如 500ms）后再验证
- **并且** 避免在输入过程中显示错误
